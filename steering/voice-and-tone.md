# Voice and Tone Guidelines

Source: AWS Content Design System & Amazon Writing Guidelines

## Voice

- Refer to the reader as "you" (second person) and refer to AWS as "we" (first person)
- For procedures or instructions, make sure the action is taken by the reader ("Now you provision the NAT instance...") rather than the author
- Reserve first person plural for speaking as AWS, with recommendations, warnings, or explanations
- Use present tense to describe current state and behavior of features and services
- Use future tense only when an event happens later than the action under discussion
- Avoid language that implies promised outcomes, such as "currently," "at this time," and "yet"

## Tone

- Direct and friendly — informal, informative, and friendly without getting chatty
- Talk to customers in their own words, never assuming they understand how services work
- Use precise technical terms where appropriate, but avoid technical jargon and insider lingo
- Speak in straightforward, plain, everyday language
- Avoid excessive words such as "please" — be courteous but not wordy

## Contractions

Use contractions carefully for a more casual tone. Use common contractions. Avoid future tense contractions.

Approved:
- With be: I'm, you're, he's, she's, it's, we're, they're
- With not: isn't, aren't, doesn't, don't, can't, couldn't, shouldn't, won't, haven't, hasn't
- With have: you've, they've, we've

Do not use:
- With will: I'll, you'll, he'll, she'll, it'll, we'll, they'll
- With not: ain't, mayn't, mightn't, oughtn't, shan't
- Archaic: 'twas, 'twere, 'tis
- Compounds with have: couldn't've
- With nouns: Amazon'll

## Modal Verbs

| Modal verb | Usage guidance |
|---|---|
| be able to | Avoid if possible |
| can | Use to express capability |
| could | Don't use to mean can or might. OK for past events |
| have to | Replace with must or need to |
| may | Avoid if possible. Don't use to mean might or can |
| might | Use to express possibility |
| must | Use to assert obligation |
| need to | Use to express needs and requirements |
| ought to | Don't use |
| shall | Don't use |
| should | Don't use to mean will or must. Use "we recommend" or "consider" instead |
| will | Avoid if you can use present tense. OK for future events |
| would | Avoid if possible |

## Requirements and Recommendations

- To be direct: use an imperative sentence ("Use the console")
- To clarify something is not optional: use "must" ("You must use the console for this procedure")
- For a gentle reminder: use "make sure" ("Make sure that you have a stable internet connection")
- For a specific requirement in context: use "need to" ("If you change the configuration, you might need to stop the instance")
- For optional suggestions: use "we recommend" or "consider" ("We recommend the following best practices")
- Don't use "should" — it can mean optional and is ambiguous
